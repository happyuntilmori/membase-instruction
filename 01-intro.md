## AI 에이전트의 "건망증" 문제

여러분이 Claude나 ChatGPT, Cursor 같은 AI 도구를 사용해봤다면 이런 경험이 있을 거예요:

> "저번 주에 내가 타입스크립트를 더 좋아한다고 말했는데, 오늘 또 자바스크립트로 코드를 써줬어..."

> "지난 달에 내 블로그 글쓰기 스타일을 설명했는데, 오늘 또 처음부터 설명해야 해?"

> "Cursor에서 저장한 내용이 Claude에서는 전혀 없는 것처럼 행동해."

이게 바로 AI 에이전트의 **건망증 문제**예요. 대부분의 AI 도구는 대화가 끝나면 모든 것을 잊어버려요. 새 대화를 시작하면 완전히 처음부터 시작하는 것처럼 행동하죠.

더 심한 문제는, **Claude는 Claude끼리만** 대화 내용을 공유하고, Cursor는 Cursor 안에서만 기억하고, ChatGPT는 ChatGPT 안에서만 기억해요. 서로 다른 AI 도구 간에는 아무 연결이 없어요.

멤베이스는 이 문제를 해결하려고 만들어진 서비스예요.

## 멤베이스가 뭔가요?

**멤베이스(Membase)**는 쉽게 말하면 **AI 에이전트들이 공유하는 중앙 기억 창고**예요.

학교에서 사물함을 생각해보세요. 내가 어떤 교실에서 공부하든 (국어 시간이든, 수학 시간이든), 사물함은 항상 거기 있고 내 물건이 저장되어 있어요. AI 도구가 어떤 것이든 (Claude든, Cursor든, ChatGPT든), 멤베이스는 항상 거기 있고 내 기억이 저장되어 있어요.

```
멤베이스 없이:
  Claude ─── 내 기억 (Claude만 앎)
  Cursor ─── 내 기억 (Cursor만 앎)  
  ChatGPT ── 내 기억 (ChatGPT만 앎)
  
멤베이스 있으면:
  Claude ───┐
  Cursor ───┼──► [멤베이스 중앙 기억 창고] ◄─── 모든 AI가 공유
  ChatGPT ──┘
```

멤베이스의 핵심 특징:

- **세션을 넘어서 기억 유지**: 오늘 대화가 끝나도 내일 기억함
- **도구를 넘어서 기억 공유**: Claude에서 저장한 것을 Cursor에서도 활용
- **플랫폼을 넘어서 기억 전달**: 어디서 작업하든 같은 컨텍스트
- **완전 자동**: 별도 저장 버튼 없이 대화하면 자동으로 기억

## 메모리 vs 위키: 두 종류의 기억

멤베이스는 기억을 두 가지 종류로 나눠서 저장해요.

### 🧠 메모리 (Memory) — 개인 컨텍스트

"나에 대한 것들"을 저장해요.

| 저장하는 것 | 예시 |
|-------------|------|
| 내 취향과 선호도 | "TypeScript보다 Python을 선호함" |
| 과거의 결정들 | "Redis 대신 PostgreSQL을 선택한 이유" |
| 반복적인 실수 | "useEffect에서 async 함수 await 빠트리면 안됨" |
| 내 목표와 마감일 | "3분기까지 사용자 1000명 달성 목표" |
| 나의 작업 방식 | "코드에 주석 달지 않는 스타일 선호" |
| 팀 정보 | "김철수 = 프론트엔드, 이영희 = 백엔드" |

### 📚 위키 (Wiki) — 참조 지식

"프로젝트나 일에 대한 문서"를 저장해요.

| 저장하는 것 | 예시 |
|-------------|------|
| API 문서 | 우리 팀 내부 API 스펙 |
| 프로젝트 명세 | 기능 요구사항 문서 |
| 아키텍처 결정 | ADR-001: 마이크로서비스 선택 이유 |
| 팀 회의록 | 2026년 1월 스프린트 회의 내용 |
| 코딩 가이드라인 | 우리 팀 코딩 컨벤션 |

위키는 마크다운 형식으로 저장되고, `[[문서이름]]` 형태의 위키링크로 문서 간 연결이 가능해요. (옵시디언 사용자라면 바로 이해할 거예요!)

## 지식 그래프란?

멤베이스는 단순히 텍스트를 저장하는 것이 아니에요. **지식 그래프(Knowledge Graph)**라는 더 스마트한 방식으로 저장해요.

지식 그래프를 마인드맵처럼 생각해보세요:

```
[나] ──── 사용한다 ────► [Python]
[나] ──── 작업한다 ────► [프로젝트 A]
[프로젝트 A] ── 사용 ──► [PostgreSQL]
[프로젝트 A] ── 팀원 ──► [김철수]
[김철수] ──── 담당 ────► [프론트엔드]
```

이렇게 **개체(Entity)**와 **관계(Relationship)**를 연결해서 저장하면, 나중에 검색할 때 훨씬 더 정확하게 관련 정보를 찾아낼 수 있어요.

예를 들어 "프로젝트 A의 데이터베이스 담당자가 누구야?"라고 물어봤을 때, 위의 그래프에서 연결을 따라가면 "김철수가 프론트 담당이고, DB는 PostgreSQL이다"라는 답을 찾을 수 있어요.

단순한 키워드 검색보다 훨씬 정확하고 스마트한 방식이에요.

## 누가 쓰면 좋을까?

### 개발자 (Developers)
- git 워크플로우, 코딩 선호도, 과거 결정들이 모든 AI 에이전트에서 유지됨
- 한 번 겪은 버그와 해결책을 두 번 다시 겪지 않음
- Cursor에서 작업하다가 Claude로 전환해도 프로젝트 컨텍스트가 유지됨

### 창작자 (Creators)
- 나만의 글쓰기 톤, 스타일, 독자 정보를 AI가 항상 알고 있음
- 과거 작품과 현재 프로젝트의 연결성 유지
- AI가 나처럼 글을 써줌

### 창업자 (Founders)
- 제품 방향, 팀 정보, KPI를 한 번만 입력하면 모든 AI가 앎
- Slack, Notion, Gmail의 결정 사항이 자동으로 메모리에 반영
- 투자자 미팅 전에 관련 컨텍스트가 자동으로 제공됨

## 가격 정책

- **무료로 시작 가능**: 신용카드 없이 무료로 사용 시작
- **무료 체험**: 14일 무료 체험 제공
- **데이터 소유권**: 사용자가 자신의 데이터를 완전히 소유. 언제든지 이의 없이 삭제 가능

최신 가격 정보는 [membase.so](https://membase.so) 웹사이트에서 확인하세요.

## 핵심 요약

- 멤베이스 = AI 에이전트들의 공유 기억 창고
